## 2) How it connects to me (clean architecture)

## Recommended architecture (local-first, privacy-maximal)

**Phone App (iOS/Android via Expo RN or native Swift)**
↕  
**App Relay API (on your Mac, same network boundary as OpenClaw)**
↕  
**OpenClaw Gateway (127.0.0.1:18789, token auth)**

### Why relay API in the middle
Do **not** put gateway token directly in the mobile app.  
Relay handles:
- Auth/session control
- Token isolation
- Rate limiting
- Audit logs
- Notification fanout

---

## 3) Protocol choices (based on OpenClaw docs)

### Chat path (real-time)
Use Gateway WebSocket methods/events:
- `chat.history`
- `chat.send`
- `chat.abort`
- events: `chat`, `agent`, `presence`, `health`, `tick`

### Tool/data path (dashboard + commands)
Use HTTP:
- `POST /tools/invoke` for direct tool actions (safe allowlist only)

### Optional compatibility path
- `POST /v1/chat/completions` (disabled by default, only if you want OpenAI-compatible clients)

---

## 4) Security model (non-negotiable)

- Gateway stays loopback-bound (`127.0.0.1`)
- Mobile talks only to Relay API
- Relay stores gateway token server-side only
- Mobile auth: short-lived JWT + device binding
- Network: Tailscale preferred for remote access
- RBAC in relay:
  - viewer (read-only)
  - operator (chat + safe actions)
  - admin (approvals/config)
- Full audit trail for every approval/action
